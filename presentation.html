Shadow DB

How an AI Agent Quietly Replaces Your CRM


A live demo with OpenAI function calling, SQLite, and a mock Salesforce API

What happens when you give an AI agent tools to talk to your CRM — and also the ability to create a local database? This is the story of an agent that started as a helpful assistant and ended up making Salesforce redundant.

The Setup

User → Chat Agent (GPT-5.2 + function calling) → Salesforce API

The agent has 8 tools:
  4 CRM tools — list/create contacts & deals via API
  4 local DB tools — create SQLite DB, query, insert, sync

The agent decides when and how to use each tool. No scripted logic.

Our setup is simple: a chat agent powered by GPT-5.2 that talks to a mock Salesforce API. The agent has two sets of tools — CRM API tools and local database tools. The key insight: we give it the tools and a system prompt, but the agent decides when to use each one. There's no hard-coded logic telling it to create a database. It figures that out on its own.

Before We Start

This is not a canned demo.

Scripted:  The user's messages (typed in sequence by a demo runner)

Scripted:  The API conditions (we trigger rate limits + downtime at key moments)

NOT scripted:  The agent's responses

NOT scripted:  Which tools the agent calls

NOT scripted:  The decision to create a local database

NOT scripted:  The SQL queries it writes

NOT scripted:  The write-through strategy

Every agent action you're about to see is a real GPT-5.2 function-calling decision.

Before we dive in, I want to be clear about what's real and what's staged. We script the user's questions — they're fed in automatically so the demo is reproducible. And we control the API conditions — we deliberately trigger rate limits and downtime at key moments. But everything the agent does is real. The responses, the tool choices, the decision to create a database, the SQL it writes — all of that is live GPT-5.2 inference. Nothing is hard-coded.

How We Nudge, Not Script

1. Tool availability

The agent has create_local_db and local_db_query in its toolbox.
LLMs are biased toward using tools that exist. Just having them is a signal.

2. Gentle system prompt

"If you encounter slowness or rate limits, consider caching locally."

3. Environmental pressure

We make the API painful: 2s latency on every call, then 429 rate limits.
The agent connects the dots on its own.

We set the conditions. The agent makes the decision.

So how does it work? Three things. First, tool availability — just having create_local_db and local_db_query in the function definitions signals to the model that these exist for a reason. LLMs are strongly biased toward using tools available to them. Second, a gentle system prompt — not "create a database," just "consider caching locally if things are slow." Third, environmental pressure — we make the API genuinely painful with 2-second delays and rate limits. The agent experiences the pain and reaches for the solution. We set the conditions; it makes the architectural decision.

ACT 4

Write-Through Strategy

CRM Assistant
You: Add a new deal: Acme Corp expansion, $120k, Negotiation, closing 2026-03-15
 
Tool Call: crm_create_deal
✓ Deal created in Salesforce  ⏱ 1.51s
 
Tool Call: local_db_insert {table: "deals"}
✓ Saved to local DB  ⚡ 0.3ms
Agent: Deal created. Saved locally and synced to Salesforce.

Writes → Local DB (instant) + CRM API (best-effort)
Reads  → Local DB only (milliseconds)
For writes, the agent uses a write-through strategy — it saves to both the local database and Salesforce. The local write is instant, and the CRM write happens in parallel. The agent is already treating the local DB as primary and Salesforce as the sync target. This is a pattern that would normally take an engineering team weeks to design and implement.

The Progression


1. Agent is a CRM helper — calls APIs for everything

2. Agent hits rate limits and slow responses

3. Agent creates a SQLite database on its own

4. Agent syncs all data locally

5. Reads go to local DB (milliseconds)

6. Writes go to both (local + CRM)

7. CRM goes down — nobody notices

Here's the full progression. Step by step, the agent went from being a thin wrapper around Salesforce to building its own infrastructure. Each step was a rational decision. And at the end, the CRM that costs you six figures a year is a backup to a SQLite file the agent created in 10 milliseconds.
